## 🧭 Как я могу помочь?

### 🐞 Сообщения об ошибках
- Используйте шаблон Issue (**Bug Report**).
- Укажите версию проекта, платформу (Vida / Forge / Fabric) и полные логи краша.
- Приложите минимальный воспроизводимый пример, если возможно.

### 💡 Предложения по улучшению
- Опишите проблему, которую решает ваше предложение.
- Обсудите идею в **Discussions** или **Discord** перед созданием Issue.
- Будьте готовы к дискуссии — не все идеи могут быть приняты в текущем виде.

### 🛠️ Разработка
- **Vida Loader** — активно развивается, приоритет у задач из Roadmap.
- **Моды** — большинство публичных репозиториев принимают исправления и оптимизации.
- **Документация** — правки опечаток, переводы, улучшение примеров всегда ценятся.

## 🔄 Процесс внесения изменений

1. **Обсудите** крупные изменения в Issue или Discord, чтобы не тратить время на то, что может быть отклонено.
2. **Создайте форк** репозитория и ветку с говорящим названием (например, `feature/new-hook`, `fix/loader-crash`).
3. **Внесите изменения**, придерживаясь стиля кода (см. ниже).
4. **Протестируйте** локально — убедитесь, что сборка проходит без ошибок, а ваши правки не ломают существующий функционал.
5. **Оформите Pull Request** с чётким описанием, ссылками на связанные Issue и пояснением сути изменений.

## 🎨 Стиль кода и стандарты

### Java / Kotlin
- Следуйте официальным конвенциям:
  - **Java**: [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html)
  - **Kotlin**: [Kotlin Coding Conventions](https://kotlinlang.org/docs/coding-conventions.html)
- Отступы: **4 пробела** (не табы).
- Имена классов: `PascalCase`, методы и переменные: `camelCase`, константы: `UPPER_SNAKE_CASE`.
- Все публичные API должны иметь **JavaDoc** / **KDoc** комментарии.
- Избегайте предупреждений компилятора — код должен быть чистым.

### Структура проекта
- Для мульти-платформенных модов используем шаблон **Architectury** или единую кодовую базу с Gradle subprojects.
- Mixin-ы должны быть минимальны и сопровождаться комментариями о причине внедрения.

### Коммиты
- Пишите сообщения на **русском** или **английском** языке.
- Формат: `<тип>: <краткое описание>`
  - `feat:` — новая функциональность
  - `fix:` — исправление ошибки
  - `docs:` — изменения документации
  - `refactor:` — переработка кода без изменения поведения
  - `style:` — форматирование, пробелы и т.п.
  - `test:` — добавление или исправление тестов
  - `chore:` — обновление зависимостей, конфигураций сборки

## 🚀 Отправка Pull Request

- PR должен быть нацелен в ветку **`dev`** (или указанную в README репозитория).
- Убедитесь, что ваш код проходит все автоматические проверки CI.
- Если ваш PR решает существующую Issue, добавьте в описание `Closes #номер`.
- Будьте готовы ответить на комментарии ревьюверов — мы стараемся давать обратную связь в течение 3-5 дней.
